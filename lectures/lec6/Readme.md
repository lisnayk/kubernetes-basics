# Лекція №6. StatefulSet та Volumes (Робота з даними)

## Робота з даними в Kubernetes: Volumes

За замовчуванням дані всередині контейнера є тимчасовими. Якщо контейнер перезавантажиться або Pod буде видалено, всі дані, створені в процесі роботи, зникнуть. Для вирішення цієї проблеми Kubernetes використовує **Volumes**.

### Основні типи Volumes

1.  **emptyDir**:
    -   Створюється під час призначення Pod-а на вузол (node).
    -   Існує до тих пір, поки Pod працює на цьому вузлі.
    -   Використовується для тимчасових даних (кеш, сортування на диску).
2.  **hostPath**:
    -   Монтує файл або директорію з файлової системи вузла (Node) в Pod.
    -   Використовується для системних завдань (наприклад, доступ до `/var/lib/docker`).
    -   **Не рекомендується** для звичайних додатків, оскільки Pod стає прив'язаним до конкретного вузла.
3.  **PersistentVolume (PV) та PersistentVolumeClaim (PVC)**:
    -   **PV** — це мережевий диск у кластері, підготовлений адміністратором.
    -   **PVC** — це запит користувача на отримання диска певного розміру та типу доступу.
    -   Це основний спосіб збереження даних (бази даних, файлові сховища).

### Динамічне виділення пам'яті (StorageClass)

Замість ручного створення PV, Kubernetes може створювати їх автоматично за допомогою **StorageClass**. Коли ви створюєте PVC, кластер дивиться на доступні StorageClasses і автоматично створює відповідний PV у хмарного провайдера (AWS, GCP, Azure) або локальному сховищі.

## StatefulSet: Робота з додатками зі станом

**StatefulSet** — це об'єкт API, який використовується для керування додатками, що мають стан (Stateful applications). На відміну від Deployment, StatefulSet підтримує ідентичність кожного Pod-а.

### Ключові особливості StatefulSet:

1.  **Стабільні мережеві ідентифікатори**: Pod-и отримують імена за шаблоном `назва-0`, `назва-1`. Ці імена зберігаються навіть після перезапуску Pod-а на іншому вузлі.
2.  **Порядковий індекс (Ordinal Index)**: Кожен Pod має унікальний цілочисельний індекс. Це важливо для систем, де потрібна чітка черговість запуску (наприклад, спочатку Master, потім Slaves).
3.  **Стабільне сховище**: Кожен Pod у StatefulSet отримує власний PVC через `volumeClaimTemplates`. Якщо Pod-0 переїде на інший вузол, він знову підключить свій старий диск `data-назва-0`.
4.  **Порядкове розгортання та масштабування**: Pod-и створюються по черзі (0, потім 1, потім 2) і видаляються у зворотному порядку (LIFO - Last In, First Out).
5.  **Headless Service**: Для роботи StatefulSet зазвичай потрібен спеціальний сервіс без IP-адреси (`clusterIP: None`). Він створює записи DNS для кожного конкретного Pod-а (`mysql-0.mysql`), що критично для внутрішньої комунікації кластера бази даних.

## Чому StatefulSet необхідний для MySQL?

Звичайний **Deployment** добре підходить для додатків без стану (stateless), як-от веб-сервери. Проте для баз даних типу MySQL він створює ряд проблем, які вирішує **StatefulSet**:

### 1. Стабільна ідентифікація для реплікації
У кластері MySQL (Master-Slave) вузли повинні знати адреси один одного.
- **Проблема Deployment**: Після перезапуску Pod отримує нове ім'я та IP-адресу. Реплікація "ламається", бо Slave не знає, де шукати Master.
- **Рішення StatefulSet**: Master завжди доступний за ім'ям `mysql-0.mysql`. Навіть після збою Slave знайде його за тим самим DNS-іменем.

### 2. Прив'язка даних до конкретного вузла
- **Проблема Deployment**: Усі Pod-и в Deployment використовують один і той самий PVC (якщо він заданий). Це призводить до конфліктів запису в одні й ті самі файли БД.
- **Рішення StatefulSet**: Завдяки `volumeClaimTemplates` кожен Pod отримує **власний** диск. `mysql-0` завжди пише у свій том, а `mysql-1` — у свій. Дані не перемішуються.

### 3. Безпечне завершення роботи
- **Проблема Deployment**: При масштабуванні вниз (scaling down) випадковий Pod може бути видалений, що може перервати активну транзакцію або пошкодити дані.
- **Рішення StatefulSet**: Видалення відбувається строго з найбільшого індексу до найменшого. Це дозволяє безпечно виводити вузли з кластера.

### 4. Послідовність ініціалізації
- **Рішення StatefulSet**: Гарантує, що Master (індекс 0) запуститься першим і буде готовий приймати з'єднання до того, як почнуть стартувати Slaves.

## Практичні приклади

### 1. Створення PersistentVolumeClaim (PVC)
Маніфест `pvc.yaml` запитує диск розміром 1 Гібібайт з доступом "ReadWriteOnce" (читання/запис одним вузлом).

### 2. Використання PVC у звичайному Pod
Маніфест `pod-with-pvc.yaml` показує, як підключити створений PVC до контейнера.

### 3. StatefulSet з MongoDB
Маніфест `statefulset-mongodb.yaml` демонструє повноцінний приклад бази даних з власним сховищем для кожного екземпляру через `volumeClaimTemplates`.

### 4. StatefulSet з MySQL
Маніфест `statefulset-mysql.yaml` та `mysql-secret.yaml` показують, як розгорнути MySQL, використовуючи **Secrets** для пароля та **volumeClaimTemplates** для збереження даних.

### 5. Deployment з PersistentVolume (Спільне сховище)
Маніфест `deployment-with-pvc.yaml` демонструє, як **всі репліки** Deployment ділять один і той самий PVC. Це зручно для читання статичних файлів, але небезпечно для баз даних.

---

## Порівняння: Deployment vs StatefulSet (Робота з PVC)

| Характеристика | Deployment (з PVC у `volumes`) | StatefulSet (з `volumeClaimTemplates`) |
| :--- | :--- | :--- |
| **Кількість PVC** | Один на весь Deployment | Окремий PVC для кожного Pod-а |
| **Спільний доступ** | Усі репліки пишуть в один диск | Кожен Pod має свій ізольований диск |
| **Імена Pod-ів** | Випадкові (хеш) | Передбачувані (індекс: 0, 1, 2) |
| **Використання** | Веб-сервери, спільні асети (Read-only) | Бази даних, черги повідомлень |

## Корисні команди

```bash
# Перегляд доступних StorageClasses
kubectl get sc

# Перегляд PersistentVolumes та Claims
kubectl get pv
kubectl get pvc

# Перегляд StatefulSets
kubectl get statefulset

# Отримання інформації про конкретний Pod у StatefulSet
kubectl describe pod mongodb-0

# Перевірка DNS-імен всередині кластера (через тимчасовий Pod)
kubectl run -i --tty --rm debug --image=busybox --restart=Never -- nslookup mongodb-0.mongodb-service

# Підключення до MySQL всередині кластера
kubectl run mysql-client --image=mysql:5.7 -it --rm -- restart=Never -- mysql -h mysql-0.mysql -prootpassword

# Перевірка спільного сховища у Deployment (зміна файлу в одному Pod змінить його в усіх)
kubectl exec deployment-pvc-HASH-1 -- sh -c "echo 'Hello from Pod 1' > /usr/share/nginx/html/index.html"
kubectl exec deployment-pvc-HASH-2 -- cat /usr/share/nginx/html/index.html
```
